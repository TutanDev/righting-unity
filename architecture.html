The 5-Layer System

Löwy's Method organizes every type by volatility — how likely it is to change. The most volatile code (UI, controllers) sits at the top; the most stable (schemas, configs) at the bottom. Dependencies flow strictly downward, one layer at a time.

LayerQuestion It AnswersVolatilityNaming
[Client]Who uses this system?Highest — UI, entry pointsNo required suffix
[Manager]What business rules apply?High — business logicMust end with Manager
[Engine]How is the work performed?Medium — algorithmsMust end with Engine
[ResourceAccess]Where is the data?Low — data accessMust end with Access
[Resource]What is the raw data?Lowest — schemas, configsNo required suffix

Client

The entry point to the system. Clients receive user input, subscribe to events, and call Managers to execute use cases. A Client should call one Manager per use case — never call an Engine or ResourceAccess directly.

LobbyClient.cs
[Client]
public class LobbyClient
{
    private readonly SessionManager _sessionManager;

    public void OnJoinPressed(string roomCode)
        => _sessionManager.JoinSession(roomCode);
}

Manager

Orchestrates business logic by coordinating Engines and ResourceAccess. Managers are the only layer that may publish events to the MessageBus. They implement use cases — one method per use case entry point.

SessionManager.cs
[Manager]
public class SessionManager
{
    private readonly ValidatingEngine _validator;
    private readonly SessionAccess _sessionAccess;

    public void JoinSession(string roomCode)
    {
        if (_validator.IsValidRoomCode(roomCode))
            _sessionAccess.Connect(roomCode);
    }
}

Engine

Performs a single algorithm or activity. Engines answer how — computation without knowing why or when it runs. An Engine must not call another Engine.

ResourceAccess

Abstracts external resources: databases, file systems, network endpoints, Unity asset bundles. A ResourceAccess must not call another ResourceAccess.

Resource

Raw data: ScriptableObjects, JSON schemas, configuration assets, external service contracts. Resources have no dependencies on other architecture layers — they are the stable foundation.

Utility (cross-cutting)

Not a layer in the enum. Types marked [Utility] can be depended upon by any layer without violating the closed architecture. The litmus test: "Could a smart cappuccino machine use this?" Logging, serialization helpers, the MessageBus — all pass.

Resolution order Type-level attribute overrides assembly-level attribute. If a type has [Manager] but its assembly is tagged [assembly: AssemblyLayer(Engine)], the type is a Manager.
Guides

Tagging Your Code

Every tracked type needs exactly one layer attribute. You can tag individual types with shorthand attributes, tag entire assemblies with [assembly: AssemblyLayer], or mark infrastructure as [Utility].

Shorthand attributes

The five shorthand attributes are the preferred way to tag individual types. Each is sealed, non-inherited, and allows only one per type.

InventoryTypes.cs
using Tutan.Architecture;

[Client]     public class HudClient        { }
[Manager]   public class InventoryManager { }
[Engine]    public class SortingEngine    { }
[ResourceAccess] public class InventoryAccess { }
[Resource]  public class ItemDefinition  { }

Assembly-level tagging

When every type in an assembly belongs to the same layer, tag the assembly once in AssemblyInfo.cs. Type-level attributes still override this for exceptions.

AssemblyInfo.cs
using Tutan.Architecture;

// All types in this assembly are treated as Engine-layer
[assembly: AssemblyLayer(ArchitectureLayer.Engine)]

// This one type overrides the assembly tag
[Utility]
internal static class InternalMathHelpers { }

Marking infrastructure as Utility

[Utility] is not a layer attribute — it's a declaration that this type is domain-agnostic and can be used by any layer. Whole assemblies can be designated as Utility via FrameworkSettings.UtilityAssemblies.

When to use Utility Logging facades, math helpers, serialization wrappers, the MessageBus, timers, audio. If the code has no business domain knowledge, it's Utility.

Marker interfaces

Optional interfaces (IClient, IManager, etc.) let you create generic constraints restricted to a specific layer. Implementing them does not assign a layer — you still need an attribute.

ServiceLocator.cs
public class ManagerLocator
{
    public T Resolve<T>() where T : class, IManager
    {
        // Only Manager-layer types can be resolved
        return default;
    }
}

Use Cases & Events

Annotate Manager methods with [UseCase] to declare use case entry points. Use [Publishes] and [SubscribesTo] to document message flow without runtime coupling.

[UseCase] on Manager methods

Apply [UseCase("name")] to public methods on Manager types. The validator traces the call chain from each annotated method through the dependency graph (BFS), validating topology and coverage.

OrderManager.cs
[Manager]
public class OrderManager
{
    [UseCase("PlaceOrder")]
    public void HandlePlaceOrder(OrderRequest req) { /* ... */ }

    [UseCase("CancelOrder")]
    public void HandleCancelOrder(string orderId) { /* ... */ }
}

[Publishes] and [SubscribesTo]

Declare which message types a class publishes or subscribes to. These are validated by the MSGBUS rules and visualized in the dependency graph. Only Managers should publish events; Managers and Clients may subscribe.

SessionTypes.cs
// Manager publishes events when state changes
[Manager]
[Publishes(typeof(SessionStarted))]
[Publishes(typeof(SessionEnded))]
public class SessionManager { }

// Client subscribes to react to events
[Client]
[SubscribesTo(typeof(SessionStarted))]
public class LobbyClient { }
How use case tracing works After tagging, open Window → Architecture → Validation and switch to the Use Cases tab. Each [UseCase] entry point is shown with its BFS call chain depth, layers visited, and any USECASE001–003 violations.

Validation Rules

Seven rule sets run after every compilation (or on demand). Each rule has a unique ID that appears in console output and the Validation window.

Rule IDCategoryWhat It Checks
ARCH001Closed ArchitectureDependencies only flow downward through adjacent layers. No skipping layers, no upward calls.
DONT001Design Don'tsClient calls only one Manager per use case entry point.
DONT002Design Don'tsManager does not call another Manager directly.
DONT006Design Don'tsEngine does not call upward into Manager or Client.
DONT011Design Don'tsEngine does not call another Engine.
DONT012Design Don'tsResourceAccess does not call another ResourceAccess.
NAME001NamingManager types end with Manager; Engine types end with Engine.
NAME002NamingResourceAccess types end with Access.
ASMREF001Assembly.asmdef hard references respect the layer topology.
MSGBUS001–007Message BusPub/sub declarations match layer roles. Only Managers publish events. Only Clients and Managers subscribe.
USECASE001Use CasesEach [UseCase] chain reaches ResourceAccess or Resource.
USECASE002Use CasesUse case call chains have no topology violations.
USECASE003Use CasesUse case chains are not orphaned (no dead-end Managers).
STRUCT001–005Structure MetricsComposition metrics: golden ratio, subsystem balance guidance.
Deliberate violation to test enforcement Tag a [Client] with a direct field reference to an [Engine] type, then run Validate Now. You should see ARCH001: disallowed dependency: MyClient [Client] → MyEngine [Engine] via Field. Fix it by introducing a Manager between them.

Settings & Enforcement

A FrameworkSettings ScriptableObject controls how violations are surfaced. Create it once via the menu and configure per environment.

Creating the settings asset

Open Window → Architecture → Create Settings Asset. The asset is created at Assets/Resources/Framework/FrameworkSettings.asset. Select it to configure enforcement mode and utility assembly names.

Enforcement modes

ModeBehaviorWhen to use
OffNo automatic validation after compilationTemporary — disabling during mass refactor
WarningsOnlyAll violations → Warning in consoleOnboarding an existing codebase
ErrorsForCoreARCH001 + DONT rules → Error; naming → WarningRecommended for active development
StrictAll violations → Error, blocks compilationCI/CD gates, pre-release hardening

Utility assemblies

Add assembly names to FrameworkSettings.UtilityAssemblies to treat entire assemblies as cross-cutting. Useful for third-party libraries (DOTween, UniTask) that every layer legitimately calls.
