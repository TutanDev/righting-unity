Optional<T>

Optional<T> represents a value that might exist (Some) or might be missing (None). It replaces null with explicit semantics and eliminates null-reference exceptions at API boundaries.

Creating Optional values

Use Some(value) to wrap a value, or None to represent absence. Both implicitly convert to Optional<T>. Some() treats destroyed Unity objects and null references as None automatically.

Example.cs
using static Tutan.Functional.F;

Optional<int> present = Some(42);
Optional<int> absent  = None;

// Implicit cast from T
Optional<string> name = "Alice";

// Returning None from a method
Optional<int> TryParseLevel(string raw)
{
    return int.TryParse(raw, out var level) ? Some(level) : None;
}

Transforming with Then()

Then() applies a function only when the value is Some, skipping over None automatically. Chain multiple Then() calls to build pipelines without nested null checks.

Pipeline.cs
Optional<Player> player = FindPlayer(id);

player
    .Then(p => p.Inventory)
    .Then(inv => inv.GetWeapon("sword"))
    .Match(
        onNone:  () => Log("No sword found"),
        onSome: sword => Equip(sword)
    );

Providing fallbacks

Use .Or(fallback) for an eager default or .OrElse(() => compute()) for a lazy one. Use .Filter(predicate) to convert a Some to None when a condition fails.

Fallbacks.cs
// Eager default
string label = playerName.Or("Guest");

// Lazy default (only computed if None)
Config cfg = maybeConfig.OrElse(() => Config.Default);

// Filter: keep only non-empty names
Optional<string> validName = rawName
    .Filter(s => s.Length > 0)
    .Then(s => s.Trim());

Pattern matching

Match(onNone, onSome) exhaustively handles both cases. It forces you to decide what to do in the absence case rather than silently returning null.

Match.cs
public Vector3 ResolveSpawn(Optional<Transform> maybeSpawn)
{
    return maybeSpawn.Match(
        onNone: () => _fallbackSpawn.position,
        onSome:  t => t.position
    );
}
Unity fake-null Unity's UnityEngine.Object equality overloads return true for destroyed objects even when the reference is non-null. Some() checks for this and returns None automatically — you don't need to guard against destroyed references.

Result<T>

Result<T> represents the outcome of an operation — either Success(value) or Error. Use it when you need to explain why something failed, not just that it did.

Creating Result values

Creation.cs
using static Tutan.Functional.F;

Result<int> ok    = Success(42);
Result<int> fail  = Error("Value must be positive");

// Wrap exceptions automatically
Result<int> parsed = Try(() => int.Parse(rawInput));

// Filter: fail if predicate returns false
Result<int> positive = parsed.Filter(n => n > 0);

Chaining operations

Then() applies a function only on success. If any step returns an error, the rest of the chain is short-circuited — you handle the error once at the end with Match().

Pipeline.cs
Try(() => JsonUtility.FromJson<PlayerConfig>(json))
    .Filter(cfg => cfg != null)
    .Then(cfg => Validate(cfg))
    .Then(cfg => Save(cfg))
    .Match(
        onError:   e => Debug.LogError(e.Message),
        onSuccess: _ => Debug.Log("Config saved")
    );

Recovering from errors

Use .Or(fallback) to provide a default on error, or .OrElse(error => compute(error)) to inspect the error and compute a recovery value.

Recovery.cs
// Fallback to default config if load fails
PlayerConfig config = LoadConfig(path)
    .Or(PlayerConfig.Default);

// Log the error, return fallback
PlayerConfig config2 = LoadConfig(path)
    .OrElse(err => {
        Debug.LogWarning(err.Message);
        return PlayerConfig.Default;
    });
Optional vs Result Use Optional<T> when "not found" is a normal, expected state. Use Result<T> when failure needs an explanation — parsing errors, IO failures, validation failures.
Guides

The Error Type

Error is an immutable value type (16 bytes on 64-bit) that carries a message and optional nested or composite inner errors. It is the failure payload inside Result<T>.

Simple error

SimpleError.cs
Error e = Error("Save file not found");

// Implicit cast from string
Result<SaveData> result = Error("Save file not found");

Nested error — wraps a cause

Use nested errors to attach a high-level message to a lower-level cause — similar to Exception.InnerException.

Nested.cs
Error inner = Error("Disk read failed");
Error outer = Error("Could not load save", inner);

// outer.Message     → "Could not load save"
// outer.InnerErrors → [{ "Disk read failed" }]

Composite error — collects multiple failures

Pass a collection of errors to create a composite that joins all messages. This is what HarvestErrors produces when multiple validators fail simultaneously.

Composite.cs
Error composite = Error(new[] {
    Error("DisplayName is required"),
    Error("Level must be >= 1"),
    Error("Currency must be >= 0"),
});
// composite.Message → "DisplayName is required; Level must be >= 1; ..."

foreach (var e in composite.AsEnumerable())
    Debug.LogError(e.Message); // logs each error separately

Validation Pipelines

A Validator<T> is a function T → Result<T>. Combine validators with FailFast (stop at first failure) or HarvestErrors (collect all failures).

Validation.cs
using static Tutan.Functional.F;

Validator<PlayerConfig> hasName  = cfg =>
    cfg.DisplayName.Length > 0 ? Success(cfg) : Error("DisplayName required");

Validator<PlayerConfig> hasLevel = cfg =>
    cfg.Level >= 1 ? Success(cfg) : Error("Level must be >= 1");

// Stops at first failure
Validator<PlayerConfig> fast     = FailFast(hasName, hasLevel);

// Runs all validators, collects all errors into a composite
Validator<PlayerConfig> harvest  = HarvestErrors(hasName, hasLevel);

harvest(config).Match(
    onError:   e => Debug.LogError(e.Message),
    onSuccess: c => Apply(c)
);

Async Support

Every operation on Optional<T> and Result<T> has async counterparts built on UniTask. Mix sync and async steps freely — the types handle the adaptation.

AsyncPipeline.cs
// Async exception wrapping
Result<SaveData> result = await TryAsync(async () =>
{
    var json = await File.ReadAllTextAsync(path);
    return JsonUtility.FromJson<SaveData>(json);
});

// Mixed sync/async chain
result
    .Then(data => data.Normalize())          // sync map
    .ThenAsync(data => Upload(data))          // async bind
    .Match(
        onError:   e => Log(e.Message),
        onSuccess: _ => Log("Uploaded")
    );
Async extensions require UniTask ThenAsync, MatchAsync, and TryAsync return UniTask<T>. Add the UniTask package to your project to use these methods.

Unity Integration

Unity-specific extension methods in LookupExtensions provide safe component access, parent traversal, and dictionary lookups that return Optional<T> instead of null.

Component lookup

HealthReader.cs
using Tutan.Functional;

public class HealthReader : MonoBehaviour
{
    [SerializeField] private GameObject target;

    private void Start()
    {
        int hp = target
            .LookupComponent<Health>()
            .Then(h => h.Current)
            .Or(0);

        Debug.Log($"HP: {hp}");
    }
}

Safe dictionary access

ItemLookup.cs
Optional<ItemDefinition> item = _itemTable.Lookup(itemId);

item.Match(
    onNone: () => ShowToast("Item not found"),
    onSome: def => ShowItem(def)
);
